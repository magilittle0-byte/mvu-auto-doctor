# MVU 自动医生（通用）

这是一个独立的 SillyTavern / TauriTavern 扩展。它包含通用 MVU 审计修复、零模型调用的正文/装备硬合同检查、分类活世界与事件连续性账本，以及不占正文的内置世界论坛，不会修改角色卡、SP·数据库、预设、缝合怪、Zsd 或 Story Oracle 文件。

## 安装

在 TauriTavern 的扩展安装页面选择“从 Git 仓库安装”，粘贴：

```text
https://github.com/magilittle0-byte/mvu-auto-doctor
```

安装后刷新一次页面。后续版本可直接使用扩展管理里的“更新”功能，不需要重新下载压缩包。

每次收到新的 AI 回复后，扩展会：

1. 动态读取当前角色卡暴露的 MVU/Zod Schema。
2. 动态读取当前启用世界书中的 `[mvu_update]` 规则。
3. 等待本轮普通 MVU 更新稳定，避免在变量仍写入时抢跑。
4. 在本地检查正文硬合同：明确字数下限、结构标签、结尾选项数、骰后自行改判、场景时间硬冲突、主动技能资源消耗，以及背包/装备对象字段与槽位合同；该步骤不调用模型。
5. 在开局窗口内检查“初始化满值、派生上限已提高、当前值却冻结”的资源失配，并用本地规则安全同步，不调用额外模型。
6. 对比上一 AI 楼层状态、当前 `stat_data`、本轮正文和原更新区块。
7. 针对规则明确带有死亡、逃跑、战斗结束、离队或失效清理条件的动态集合，按当前条目名定向回查历史证据；不会因为近期没提到就擅自删除。
8. 让模型同时审计错更、漏更、动态成员资格、输入变化后的依赖闭包、装备/效果转移双方，以及规则能够唯一证明的正文硬错误；变量补丁和正文修正版共用同一次模型调用。
9. 在内存副本上校验路径、操作、MVU 解析和 Zod 结果。
10. 全部通过后才原子写入，并在写入后回读验证；普通修复、开局同步和撤销共用同一写入队列，不能并发踩同一楼层。
11. 把变量纠错块持久附加在当前 swipe 中；若正文也有可证明硬错误，则另建一个可左滑撤回的修正版 swipe，并复制修正后的 MVU 快照，不重放资源 delta。
12. 刷新消息与 `<StatusPlaceHolderImpl/>`，让正文状态栏立即重建。

v1.2.0 开始，每次任务还会绑定聊天 ID、稳定消息 ID、楼层、swipe、正文指纹和生成世代，并在每个异步步骤后复核。数据库预热/算 Token 的 `dryRun` 会被忽略；普通生成、swipe、regenerate 与 continue 分开记录。真正写变量时使用精确楼层而不是笼统的 `latest`，最大限度避免切聊天、重抽或连续生成时落错位置。

模型连接顺序：

- 默认优先使用 Story Oracle 已配置的连接（稳定 Hook API v1）。
- Story Oracle 未安装或兼容接口不可用时，使用酒馆当前主连接的 `generateRaw`。
- Story Oracle 已经实际发起请求后若遇到 429、鉴权、网络或服务端错误，不会再静默切到可能指向同一公益站的主连接撞第二次，也不会把连接错误当成分析失败重试。

正常自动模式下，每条 AI 回复的调用构成是：硬合同本地检查 0 次；变量诊断成功时 1 次；活世界整理 1 次；内置论坛默认为手动，因此 0 次。正文修正版合并在变量诊断的同一次返回里，不会再调用一次模型。只有变量模型返回空结果、缺失/截断区块、坏 JSON、非法补丁或未通过本地校验时才定向重试，整轮最多 3 次；不是每轮固定调用三次。429、鉴权、连接和服务错误不重试。坏世界账本或论坛批次仍各最多重试 1 次。

v1.8.0 起，设置页的操作时间线和悬浮面板“工具”页会按当前聊天显示模型调用总数，并分别统计变量、活世界、论坛、失败和 429。统计只保存计数和时间，不保存正文、提示词、API 地址或密钥；切换聊天后读取该聊天自己的计数。

变量诊断默认使用 `32768 max_tokens`，不再强压到 4096；设置页“变量诊断模型适配与输出空间”可以改成所用模型/公益站实际允许的单次输出上限。医生首先完整输出 `<UpdateVariable>`，然后才允许追加可选正文修正版；补丁 JSON 数组完整但中转遗漏尾标签时会本地安全恢复，不浪费一次调用。正文修正版损坏只会被忽略，不再连带否决已经通过校验的变量补丁。

同一设置区还提供“附加破限/模型适配提示词”。医生仍自动构造 Schema、`[mvu_update]`、当前状态、初始化基线、正文、上下文和机器输出协议；这里只需粘贴模型适配的几句文字，留空即使用默认。

为避免两个自动程序同时写变量，扩展默认会关闭 Story Oracle 的 `AUTO` 诊断开关；Story Oracle 的手动诊断不受影响。
如果检测到兼容的 Story Oracle 仍开启 `AUTO`，但该设置不可写或无法通过第二次独立读取验证为关闭，自动医生会拒绝普通修复与开局资源写入，而不是冒险双写；正式提交 MVU 前还会再复查一次，支线账本和独立论坛仍可继续运行。

每次候选补丁除了核对明确触碰的路径，还会验证原状态中所有未触碰叶子字段仍被完整保留。解析器或新版 Schema 若意外剥离旧卡遗留字段，候选会在写入前被拒绝；新增的 Schema 默认字段仍允许保留。以 `_` 开头的只读/派生字段只允许在仍然存在时由 MVU 归一化重算，不能被补丁触碰或被解析器删除。

若普通 MVU 长时间保持“正在额外解析”或连续读数无法稳定，医生会以 `busy` 安全跳过本次任务，不调用修复模型、不写变量。若写后回读失败，回滚只恢复本次补丁实际触碰的路径，保留其他插件或普通 MVU 同期产生的合法状态，并以语义子集而非整树字符串指纹对回滚结果再次确认；宿主重排键或补充 `display_data`/版本字段不会造成假失败。

每次真正写 MVU 之前会先保存带快照、触碰路径和前后路径值的恢复记录，并等待宿主的 `saveMetadata` 或 `saveChat` 完成后才调用 `replaceMvuData`；只有不可等待的防抖保存接口时会安全取消变量写入。写入通过回读后把同一记录升级为已应用，最后才尝试把纠错块附加到原 swipe。即使用户恰在异步写入飞行中切换聊天/swipe，已经落地的变量变化也不会被误报成“未写入”，新回复不会被读取或刷新，回到原目标后仍可核验并撤销。撤销只恢复该次修复触碰的路径，其他插件后来新增的显示、版本或无关状态继续保留；若写入根本没有落地，预备记录会识别为无需撤销。正文刷新成功后，同一日志记录才标记为已同步。

## 必须这样设置

- 安装后先打开设置页顶部“环境自检”。它会只读检查 MVU API、故事神谕 AUTO、模型连接、活世界注入通道和上轮注入是否真正落入最终提示词；自检不会替你打开额外解析或修改角色卡。
- 关闭 MVU 的“额外 AI 解析/额外模型更新变量”。本扩展跟随角色卡原本的 `<UpdateVariable>` 更新完成后再审计，不需要第二个 MVU 解析模型。
- 只保留一个自动修复器：开启“MVU 自动医生”，关闭 Story Oracle 的 `AUTO` 诊断。Story Oracle 的手动诊断、问剧情、校正回复、世界书模式仍可正常使用。
- SP·数据库继续使用作者原版填表提示词。自动医生不读写数据库表，也不接管数据库回忆。
- 同一角色只启用一套当前版本的 MVU 更新规则。若作者卡已内嵌世界书，不要再同时启用一套重复旧世界书。

“关闭 MVU 额外 AI 解析”只针对 MVU 自带的第二次变量解析。活世界与事件连续性开启时，会在主回复落地后使用插件连接整理一份独立的因果账本，但它不写 MVU，也不代替角色卡原本的变量更新。

## 开局资源初始化修正

设置中的“开局自动补满初始化失配的资源”默认开启。它用于修复这类角色卡错误：世界书把资源初始化为满值（例如 MP 50/50），角色创建完成后前端把上限派生为 110，却只留下当前值 50。

- 只在当前聊天最前面的 4 条 AI 消息内运行，包括角色卡开场白。
- 优先读取当前激活世界书的 `[initvar]`/“变量初始化”条目；存在外置条目时以其为权威，不再混用卡内嵌副本。没有外置条目时读取卡内嵌初始化；两者都不可读时，才用上一楼确实满值的状态作证据。
- 不写最大值，只把已证明属于初始化失配的“当前值”同步到现有最大值。
- 明确 initvar 满值证据不依赖 HP、MP 等专属字段名，会识别中英文“当前/最大/上限”成对字段，因此可用于其他 MVU 卡；只有缺少 initvar、退回“上一楼满值”单证据时才要求字段具备通用资源语义。
- 初始化不是满值的容量字段不会动，例如负重 0/55；本轮更新区块明确改过的资源也不会动，因此不会把战斗消耗或主动施法误判成开局补满。
- 同时识别 `MP_当前/MP_上限`、`HP当前值/HP最大值`、`MP:{当前,上限}` 和数组内角色资源等常见通用结构；YAML `- 名称: 术士` 块序列及 `- 属性:` 下的嵌套映射会按缩进解析为数组对象，嵌套映射后的同级字段不会被误吞，initvar 数字后的行内注释也不会把数值误解析成字符串。所有形态仍须满足可验证的开局满值证据，`0/0` 视为尚未建立正上限，不会在派生为 `0/5` 时自动补满。
- 仅凭上一楼满值推导时会排除任务进度、阶段、次数等非资源计数，避免把 `3/3 → 3/5` 的新阶段误补成完成；只有明确 initvar 证据才能覆盖这项保守排除。
- 多阶段捏人再次提高上限时，只要该资源仍保持上一步同步后的满值，可以在开局窗口内继续同步。
- 每次修正同样经过 MVU 解析、Schema、精确楼层写入、写后回读和可撤销记录，不是仅修改前端显示。

## 正文与装备硬合同

硬合同检查读取当前启用预设、世界书、角色卡 Schema、最新 AI 回复、上一条用户 `<scene>` 与实时 `stat_data`。本地阶段只执行能够由程序确定的检查：

- `<content>`、`<options>`、`<UpdateVariable>`、`<JSONPatch>` 等结构是否闭合且 JSON 合法。
- 规则或规划明确声明正文区间时，统计 `<content>` 内汉字；左值是最低验收线，右值只是默认写作目标，不会把自然略超目标的完整因果段落误报为错误。
- 明确要求四个结尾选项时，只检查选项数量。
- 骰后锁是否把未消费骰源的批量结果直接“设定为成功”，或在唯一骰源合同下自行重投。
- 上一条用户 `<scene>` 的明确时间戳与 `A·S0` 或 V5 `A·边界` 中的 S0 起始时间是否冲突。
- 正文明示发动技能后，规则中声明的 MP/HP/耐力等资源是否足额扣除。
- 背包物品是否满足当前卡声明的描述/数量等字段，背包装备字段是否残缺，以及 Schema 声明槽位标签后实际装备位置是否匹配。

这不是“文风审稿器”，也不是剧情裁判。合理的额外战利品、NPC 反应、环境细节、惊喜、支线延伸和其他 GM 自主发挥不会因为玩家未逐项指定就被报警；检查器不评价措辞。

v1.7.0 起，确定性硬错误会并入原本那一次变量诊断，模型可同时返回最小正文修正版，不新增第二次模型等待。字数不足时只能补原行动 A 的过程/锁定结果、NPC 基于有限信息的自主反应以及环境和后果；不得替玩家新增 B/C/D 行动、对白、移动、技能、资源消费或检定。应用前还会运行玩家新增动词/对白/技能守卫，并再次本地审计；原文、骰面、成功等级、思维链、S1 和变量更新块保留在旧 swipe，可随时左滑恢复。

对任意卡特有的掉落公式、奖励数量或物品格式，模型必须逐字引用当前 Schema、世界书规则或预设合同中的短证据；本地确认该证据真实存在、修正版未引入新硬错误，并且变量补丁也有对应变化后才允许落地。无可核验证据时只报告或修变量，不凭常识改正文。

当前无限回廊 5.4.1 实跑状态暴露出一个上游合同缺口：装备槽位只由 `装备.头部/躯干/主手/...` 的当前路径表示，装备对象本身没有 `装备位置` 或 `可装备槽位`；物品退回背包后，程序无法无歧义知道应该装备到哪里。医生会把它标记为 Schema/世界书合同提醒，但不会凭空猜槽位。真正修复需要在角色卡 Schema、装备生成规则和穿戴前端中统一增加并校验同一个槽位字段；仅改医生不能让旧前端禁止“手枪装到头部”。

## 活世界与事件连续性

该功能不会强行在正文末尾公开所有幕后事件，而是把状态保存在当前聊天的插件元数据中，并在下一回合选择性注入真正影响 RP 的公开状态。v1.6.0 起，医生在原有连续性账本之上增加事件时钟和分类世界快照；“平行事件”不是正文衍生支线的同义词，而是四类并存的活世界事件：

推进结构参考了 [DlSNlGHT/World](https://github.com/DlSNlGHT/World) 的实际源码机制，包括事件阶段内进度、风声衰减、稳定 ID 增量合并和条件筛选注入；本扩展不依赖或安装该项目，也不沿用其题材化文案和复杂配置界面。

- `主线衍生`：由正文中已经发生的行动、人物或因果直接产生。
- `世界设定·暗中相关`：尚未在主线露面，但按世界设定存在潜在因果，等待真实传播链或接触条件。
- `世界设定·当前独立`：依据角色卡/当前世界书独立发展，可能很久以后自然相交，也可以永远不相交。
- `世界脉动`：组织、社会、生态、日常生活或局势在玩家视野外自行变化，可以在幕后自行结束。

运行规则：

- 识别预设的 `<parallel_event_record>`、`<current_event>`，以及缝合怪的 `<dm_story>`、`<npc_track>`。
- 给事件保留稳定 ID、冲突/事务类型、1–4 级规模、阶段内 `1/9` 至 `9/9` 进度、来源类型、因果父项、与主线关系、幕后变化、触发条件、下一拍、自然汇流条件、知情层级、持续影响、流言与设定依据。
- 每个完成的 AI 回合都会先由本地事件时钟为所有未结事件给出“推进／保持／受挫”基线；冲突等级越高越容易升级，事务等级越高越难完成，连续不成功达到上限后会保底推进。模型再依据真实能力、资源、信息、距离和阻力复核，可标记暂时受阻或按明确因果终结，不会为了机械推进凭空补条件。
- 若切换页面、长生成或插件队列让若干 AI 楼层尚未落账，下一次调度会按 `lastSource` 一次补记所有漏掉的完成回合，而不是只处理最新一楼；旧 v2 账本会继续迁移使用。
- 当世界模型遇到 429、超时、空响应或坏 JSON，本地事件/传播时钟已经产生的变化仍会保存并明确标记降级，不再把整轮回退成“未产生有效节拍”。
- 一轮仍只允许一条旧事件产生新的实质叙事变化；其他事件只更新时钟压力，不要求全部进入正文。若正文只过去片刻、时间窗未到或因果前提未满足，可以登记带有具体依据的 `held`。空泛的“无变化”不算有效调度。
- 玩家本轮 A 行动直接造成持续 B 影响时，可以在推进旧事件的同一回合登记 main_derivative 事件 C；不再发生“推进旧事件就丢掉新因果”的二选一。
- 事件结束时必须登记结束方式和至少一项持续影响或流言。仍会自行发展的 D 后果使用 `causedBy` 派生新稳定事件；旧事件保持 resolved，避免把同一支线无限续命。
- 预设反复输出同一平行事件标记时，只更新真正出现的新事实；既有结束方式、持续影响、流言、知情层级、紧迫度和创建回合不会被空默认值抹掉。
- 每轮最多让一条旧事件改写叙事事实、最多接纳两条有正文或父事件证据的因果新事件；无直接因果的世界脉动仍遵守自主度节流，避免事件爆炸。
- 分类世界快照分为长期趋势、势力、风声、声誉、环境、隐秘和因果联动。声誉使用“机构／公众／地下圈层／专业圈层”，所有名词与例子取自当前角色卡和世界书，不预设古代、现代、赛博或奇幻文案。
- 世界模型只返回本轮有实质变化的字段；本地按稳定 ID 增量合并，未返回的旧势力、风声或隐秘不会被整块覆盖。新条目由本地分配 ID，模型编造的未知 ID 会被拒绝。
- 世界推演按“私密性 → 长期趋势 → 新风声 → 传播节点 → 覆盖圈层 → 跨类别影响 → 声誉/敌情/经济”的顺序检查。无目击、未留痕迹的私密行为只留在隐秘分类，不会直接变成公共声誉或让不知情 NPC 行动。
- 不同类型的风声有不同沉寂宽限与衰减速度；只有出现新的合法传播节点才能扩大强度或范围。跨类别变化必须登记因果联动，经济和声誉单轮最多移动一级。
- 下一回合只注入已公开或已经影响当前局势的分类状态；隐藏条目、低传播风声与尚未汇流的独立事件仍留在审计面板。
- hidden / rumor / observed 分层，禁止让不知情角色突然全知。
- 只推动 NPC、势力、敌方、环境、约定与离场角色；禁止替玩家角色行动、回答、移动、消费资源或追加检定。
- 检测到预设或缝合怪时采用桥接模式，由原系统保留剧情提案权；插件只做接续、去重和回收。
- 缝合怪的成功/失败路线保留为条件式导演提案，不作为裁决目标。先按当前骰子前端的固定位置或顺序消费唯一骰值并结算 DC/成功等级，再选择对应剧情分支；禁止浏览骰池挑成功数字或先写结果后补造检定。
- 检测到 DlSNlGHT/World 世界引擎时同样进入桥接模式：世界引擎保留事件链、风声与世界推演提案权，变量医生不再把自己当唯一导演，只补缺失因果并避免同一事件重复推进。
- 自主事件只从当前角色卡与当前启用世界书的叙事设定取材，可创建符合设定的普通 NPC、小组织和地方事务，但不能凭空改写核心世界观或重要角色过去。
- 独立事件默认只在幕后账本推进，不为了展示伏笔强塞进正文；只有传播、因果、人物接触、地点或时间条件真实满足时，才先进入“正在汇流”。
- hidden 且尚未汇流的事件向主回复只暴露稳定 ID、阶段和调度边界，连语义化标题也会折叠；最近调度的 `lastTick.reason` 同样替换为不含幕后事实的通用说明，不能绕过标题折叠。事件在幕后结束后，其标题、结束方式、持续影响与流言也继续留在后台账本，直到知情层级或真实汇流条件允许公开。标题与幕后细节只保留在玩家主动打开的审计视图中。
- 独立事件允许一直不与主线相交，也允许在幕后解决。结束后的影响和有传播链的流言会继续保存在世界状态中并可派生后继事件；只有公开条件满足后才进入主回复，插件不会把所有变化都变成围绕玩家的任务。
- 已结束事件进入独立归档容量，不再占用活动事件槽位；长期游玩不会因为历史事件塞满上限而停止生成。

模式说明：

- `自动活世界（推荐）`：有角色卡/世界书叙事设定时即可在回复后维护活世界；检测到预设/缝合怪时自动桥接去重。
- `始终开启`：没有结构化标记时也整理正文中的明确未决因果。
- `关闭`：撤下注入，不运行世界整理；已有账本仍保留，除非手动清空。

自主度说明：

- `保守`：只登记正文、预设或缝合怪已经提出的因果，不主动建立幕后事件。
- `活世界（推荐）`：自主新事件至少间隔 3 个账本轮次，最多保留 3 条未结自主事件。
- `活跃`：自主新事件至少间隔 2 个账本轮次，最多保留 4 条未结自主事件。

重抽或重新生成时，正文生成阶段使用该楼回复产生前的世界/事件存档点；新 swipe 落地后再从存档点重算，不让被放弃的旧 swipe 污染新剧情。

只在已有 swipe 间手动切换、没有重新生成时，也会监听宿主的 `MESSAGE_SWIPED` 并无模型回退到该楼生成前存档点；若宿主版本没有发出该事件，下一次生成开始时还会用账本/论坛的 `lastSource` 复核，在把连续性注入正文前完成同样回退。hidden 且尚未显现、未汇流的 independent/latent 事件不能仅凭模型声称升级成公开传闻，论坛取材处还有第二层本地闸门。

当前候选为每个最新 AI 楼层保留一个“整楼生成前”世界/事件/论坛存档点，而不是保存该楼每个既有 swipe 的完成后账本快照。因此，手动切回一个已经看过的旧 swipe 时会保守回到整楼生成前状态，防止另一分支的事件或帖子串入；它不会即时还原那个旧 swipe 当时生成后的账本。后续真正生成/续写时会按当前分支重新结算。需要逐 swipe 浏览历史时，应等未来引入按稳定 swipe 身份保存的有界快照，不能复用另一分支结果冒充。

活动线程超过设置容量时不会再被截断删除：超出的未结因果转为 `dormant`，保留触发条件、因果链和审计卡片，并在界面显示休眠保留数量；提高容量或调度条件成熟后可以重新唤醒。事件仍只有在登记 resolution 与持续影响/流言后才进入 resolved。

## 手机前端

设置页、贴边悬浮球、世界/事件/论坛/工具四分页和完整论坛共用同一套主题色、状态条、圆角与触控规范。世界页先显示一段摘要，再把长期趋势、势力、风声、声誉、环境、隐秘和因果联动分别放进默认收起的分类栏；事件详情留在独立页，摘要只显示阶段与紧迫度，展开后按“当前 / 因果 / 传播与收束”分组且隐藏空字段。设置页各功能区默认折叠，完整事件账本只在悬浮面板保留一份。悬浮球贴到左右边缘后，计数角标会自动翻到屏幕内侧，不会被手机边缘裁掉；长按约一秒可把位置恢复到默认值。

## 世界论坛

“打开独立论坛”可以直接打开扩展自带的论坛，不要求安装 Zsd。论坛以当前聊天为单位保存帖子、评论、版块、热度与刷新页次，并使用独立模型请求增量刷帖，不占正文篇幅、不写 MVU、数据库或事件账本。v1.6.0 起默认改为手动刷新：论坛标题栏始终保留“刷新论坛”按钮；来源、手动/自动切换、自动间隔、Zsd 入口和清空操作放在默认收起的“来源与管理”中，不再占据手机阅读首屏。

- 首次刷新生成 4 至 5 个帖子和 6 至 12 条评论，每个首屏帖子都至少有一条回复；后续每次生成 2 至 4 个新帖和 6 至 12 条评论，并优先给零回复旧帖补楼。
- 至少一半内容是日常闲聊、求助、攻略、交易、吐槽、八卦或地方话题；不同网友会补充、质疑、开玩笑或跑题，不把整个论坛变成任务板。
- 生成结果若帖子或评论数量不足、首屏仍有孤帖、后续没有补零回复旧帖，会自动拒绝并重试一次，不把不合格的一页保存到聊天。
- 只把 observed / rumor 等已经具备公开传播条件的世界风声交给论坛；hidden 幕后事件和私密正文不会泄露给网友。
- 论坛模型不直接接收最近正文或完整世界书；世界设定只允许带有公开、常识、地理、交通、制度、生活、新闻等公共语义的条目进入取材池，标题、关键词或正文含秘密、幕后、真相、未触发和剧透信号的条目都会在本地过滤，中英文 `GM/DM eyes only`、`do not reveal` 等写法同样拒绝。
- 模型偶尔漏写数组/对象项之间的逗号或多写末项尾逗号时，本地会在不改变任何字段内容的前提下安全修复标点；修复成功会明确显示。输出被截断、缺字段值或结构不确定时仍会拒绝整批，不会把半截论坛写入聊天。
- 普通帖子永远留在论坛。只有帖子已经造成持续的聚集、传播、短缺、人物决定或外部行动时，下一轮支线调度才可把它登记为因果事件；网友猜测不能倒推成真相。
- 支线产生公开风声后可以进入论坛；论坛形成真实公共行动后又可在下一轮回到支线账本，再通过连续性注入自然影响正文，形成闭环而不是三套系统各写各的。
- 默认手动刷新；普通 AI 回合和打开空论坛都不会暗中调用论坛模型。点击论坛标题栏或设置页的“刷新”才生成一页。
- 可切换为按 AI 回合自动刷新，并把间隔设为 1 至 12 回合；旧版自动刷新配置升级到 v1.6.0 时会安全迁移为手动，避免升级后继续产生用户未预期的调用。
- 重抽会使用该楼刷新前的论坛存档点，旧 swipe 生成的帖子不会污染新回复；切换聊天后只显示新聊天自己的论坛。
- 同一楼层使用 continue 续写时保留整楼生成前的论坛与支线存档点；之后 regenerate/swipe 仍从整楼开始前重算，不会把续写中间态带进新分支。
- 每段完成的 continue 仍各运行一次世界节拍；论坛只有在选择自动模式时才随 continue 刷新。整楼前存档点不会因此被覆盖。
- 默认来源为“医生内置论坛”；安装 Zsd 不会替换数据源。主动切换为“Zsd 论坛”后，悬浮入口改为打开 Zsd，医生内置帖子仍保留但不刷新。

### 手机端查看活世界账本

安装后默认出现一个写有“脉”字的悬浮球，也仍可在“扩展设置 → MVU 自动医生（通用）→ 活世界与事件连续性”查看同一份账本：

- 悬浮球可上下拖动，松手后吸附到最近的左右边缘，闲置时只露出一小段；点击即可拉出并打开面板。位置会保存在本机。
- 角标显示当前未结事件数；整理中、成功或失败会使用不同的光效提示。可在扩展设置关闭悬浮球，状态栏入口始终保留。
- 悬浮面板分为“世界 / 事件 / 论坛 / 工具”四个页签并记住上次所在页。世界页内部再按类别折叠，事件卡和工具不会与世界摘要纵向堆叠。
- 面板提供“检查变量”“检查硬规则”“整理世界”和论坛入口。论坛入口会遵循你选择的来源；选择内置论坛时进入移动端完整论坛，空页面会等待用户手动刷新。
- 如果检测到 Zsd，内置论坛会显示兼容提醒和“打开 Zsd”快捷按钮；是否暂停内置自动刷新只由可见的“论坛来源”选项决定。

- 顶部显示未结数、已收束数、账本轮次、更新时间和当前来源模式。
- 未结事件按“已显现 → 推进中 → 已埋设 → 搁置”及紧迫度排序；第一条不含剧透的事件默认展开。
- 尚未在主线显现的独立事件默认只显示“幕后独立事件（点击查看剧透）”，避免玩家提前失去惊喜；可关闭剧透折叠。
- 点击每条事件可查看来源、因果父项、与主线关系、设定依据、当前进展、幕后变化、结束方式、持续影响、传播中的流言、下一自然接口、触发条件、汇流条件、人物地点、知情范围和最近来源楼层。
- 已收束事件放在单独的折叠区，不与活动事件混在一起。
- “风声”分类只汇总已经形成传播链的消息。普通水帖、吐槽、玩笑和日常互动不是世界事件，继续由卡内贴吧或独立论坛生成。
- “刷新显示”只重读当前聊天元数据，不调用模型；“立即整理世界”才会调用模型更新账本。
- 该区域是玩家审计视图；只有你主动展开剧透卡片时才显示角色尚不知情的幕后信息，插件不会因此把隐藏知识泄露给正文角色。

切换聊天、收到新回复、整理完成、重抽或清空账本后，显示会自动刷新。选择“关闭”只停止后续整理，旧账本仍可查看。

扩展设置里提供：

- 自动开关
- 零模型调用的正文/装备硬合同自动检查、手动复查与问题明细
- 开局初始化资源满值同步开关
- 模型连接优先级
- 变量诊断单次输出上限与附加破限/模型适配提示词
- 双写保护
- 回复后等待时间
- 手动检查最新回复
- 跨刷新保存、并经过变量指纹保护的撤销记录
- 活世界的自动兼容/始终开启/关闭模式
- 活世界自主度与幕后事件剧透折叠
- 手动整理世界与清空当前聊天账本
- 手机端可展开的事件账本明细与已收束归档
- 可拖动、贴边半隐藏的悬浮球，以及不占正文的完整内置世界论坛
- 论坛来源、手动/自动刷新切换、自动刷新频率、Zsd 双生成提醒和当前聊天清空

插件不包含任何角色卡专属字段、路径或枚举。角色卡更新或切换到其他 MVU 卡时，会重新读取新卡的规则和 Schema。

聊天元数据只保存有限条修复快照、活世界/事件账本、内置论坛帖子和来源指纹，不保存 API 密钥或完整模型提示词。单条 MVU 快照超过安全体积时不会写入聊天，此时该条修复仍然有效，但无法跨刷新撤销；日志仍会保留该记录，重载后点击撤销会明确说明“快照过大”，不会假装没有发生过修复。
